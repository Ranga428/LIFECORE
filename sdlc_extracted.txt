AI-Powered Life Tracker
Master Software Development Life Cycle Plan
1. Document Purpose
This is the central reference document for the LifeCore development project. It maps the complete Software Development Life Cycle (SDLC) from inception through maintenance, providing a navigational hub that connects all seven phase documents.
As a solo developer building a complex, AI-powered application, this document ensures that every phase is executed with the discipline of a full engineering team — leveraging IEEE standards adapted for a single-person workflow.

4. Document Series Map
4.1 How to Use This Series
Each document in this series is self-contained but cross-referenced. As a solo developer, use this center document as your daily navigation tool to know which doc to consult at each stage of work.
5. Solo Developer Strategy
5.1 Adapted IEEE Standards
IEEE standards are designed for teams. The following adaptations make them practical for a solo developer without sacrificing rigor:
SRS reviews are done as self-review sessions scheduled at phase boundaries, not committee reviews.
SDD architecture decisions are logged in an Architecture Decision Record (ADR) file instead of formal stakeholder meetings.
Test plans execute sequentially rather than in parallel — prioritize critical paths and AI-dependent features first.
Maintenance documentation is updated incrementally with each release rather than at planned review cycles.

5.3 High-Level Timeline
Estimated durations for a solo developer working part-time (~20hrs/week). Adjust based on your actual availability.
6. Key Principles & Commitments
6.1 IEEE Standard Commitments
Every feature in the SRS will have a corresponding test case in the STD — no untested requirements.
All design decisions in the SDD will trace back to a specific SRS requirement.
The test summary report will include a formal pass/fail assessment before any production release.
Risk registers will be maintained and reviewed at the start of each new phase.
6.2 AI-First Development Principles
AI features will be designed with graceful degradation — the app must remain functional if AI services are unavailable.
All AI-generated content will be user-editable; the AI assists, the user decides.
Privacy by design: no user data is sent to AI providers without explicit consent and clear disclosure.
Agentic AI workflows (workout plans, tracker generation) will include preview + confirmation steps before executing.
6.3 Quality Gates
Each phase must meet these gates before the next phase begins:
Planning → Requirements: Project charter approved by developer, tech stack finalized
Requirements → Design: All features have priority ratings and acceptance criteria
Design → Development: Architecture decision record complete, DB schema validated
Development → Testing: All features marked 'done' in sprint board, build deploys successfully
Testing → Deployment: Test summary report shows pass on all P0/P1 test cases
Deployment → Maintenance: Production monitoring active, rollback plan documented

=== 02_Requirements_SRS.docx ===
Phase 2: Requirements
Software Requirements Specification (SRS) — IEEE Std 830
Document 02 of 07  |  LifeCore AI Life Tracker
1. Introduction
1.1 Purpose
This Software Requirements Specification describes the functional and non-functional requirements for LifeCore version 1.0. It is the authoritative reference for all development and testing activities.
1.2 Scope
LifeCore is a cross-platform mobile and web application enabling users to manage their entire life — scheduling, fitness, nutrition, finance, medicine, wellness, and productivity — through AI-powered interfaces. The app targets individuals, students, professionals, and enterprise teams.
1.3 Definitions
2. Overall System Description
2.1 Product Perspective
LifeCore is a standalone application that integrates with external services (Google Calendar, LLM APIs, nutrition databases, finance APIs) via adapters. It does not depend on any specific third-party platform for its core functionality.
2.2 User Classes
Individual User: Primary user managing personal life domains.
Power User (Premium): Access to all AI features, unlimited templates, priority processing.
Enterprise Admin: Manages group accounts, dashboards, and team-wide templates.
Enterprise Member: Team member with access to shared group trackers.
2.3 Operating Environment
iOS 15+ (iPhone and iPad)
Android 10+ (phones and tablets)
Modern web browsers: Chrome 100+, Safari 15+, Firefox 100+
Backend: Linux-based cloud infrastructure (Supabase / Vercel)
3. Specific Requirements
3.1 AI Life Tracker Builder
FR-01: One-Prompt Tracker Generation
The system shall accept a natural language prompt describing the user's life goals and generate a complete, personalized tracker dashboard within 10 seconds.
The generated tracker shall include relevant sections based on detected intent (fitness, nutrition, schedule, finance, etc.).
The user shall be presented with a preview before the tracker is applied.
The user shall be able to edit any AI-generated element before confirming.
FR-02: Tracker Editing via AI
The system shall allow users to modify their tracker by issuing natural language edit commands (e.g., 'Add a water intake tracker').
3.2 Smart Scheduling & Timer System
FR-03: Event Scheduling
The system shall allow users to create, edit, and delete scheduled events with title, date, time, duration, and recurrence settings.
FR-04: Automatic Timer Generation
The system shall automatically generate countdown timers for all scheduled events. Timers shall notify the user 15 minutes before, 5 minutes before, and at event start.
FR-05: Customizable Morning Alarm with AI Snooze
The system shall support morning alarms that, when snoozed, prompt the user with a configurable AI question (e.g., 'Why are you snoozing?'). The user must answer before the snooze is granted.
3.3 Google Calendar Integration
FR-06: Calendar Sync
The system shall sync with the user's Google Calendar account via OAuth 2.0. Events created in LifeCore shall appear in Google Calendar and vice versa within 2 seconds of creation.
FR-07: Temporary Schedule Adjustment
The system shall allow temporary schedule adjustments that modify the display view in LifeCore without permanently altering the underlying Google Calendar event.
3.4 Fitness & Workout Planning
FR-08: AI Workout Plan Generator
The system shall generate personalized workout plans from a user prompt (e.g., 'I want to lose 10 lbs in 3 months with 4 workouts per week'). The plan shall include exercises, sets, reps, rest periods, and weekly progression.
The workout generator shall use an agentic workflow with a preview-and-confirm step.
Generated plans shall be editable by the user.
3.5 Nutrition & Grocery Management
FR-09: Grocery List with Macro Calculation
The system shall allow users to create grocery lists and display estimated macronutrients (calories, protein, carbohydrates, fat) per item and per list total.
FR-10: Food Photo Analysis
The system shall allow users to photograph a meal. The AI shall identify food items in the photo and estimate their caloric and macro content. Accuracy confidence shall be displayed.
Users shall be able to correct AI estimates manually.
Photo analysis results shall be logged to the user's daily nutrition tracker.
3.6 Finance Manager
FR-11: Budget Tracking
The system shall allow users to set monthly budget categories (rent, food, entertainment, etc.) and log income and expenses. The system shall display current vs. budgeted spending with visual indicators.
FR-12: Financial Insights
The system shall provide AI-generated weekly financial insights summarizing spending patterns and suggesting adjustments.
3.7 Medicine Tracker
FR-13: Medication Schedule
The system shall allow users to add medications with dosage, frequency, and start/end dates. The system shall send push notifications at the scheduled time.
Users shall be able to log dose taken, skipped, or delayed.
The system shall maintain a 30-day medication history log.
3.8 Audio Transcription Note Taker
FR-14: Voice Recording & Transcription
The system shall allow users to record audio notes. The audio shall be transcribed to text using AI (Whisper API) with a processing time of under 5 seconds for recordings up to 60 seconds.
Transcriptions shall be editable and searchable.
Users shall be able to tag notes by category.
3.9 PDF Book Reader
FR-15: In-App PDF Viewer
The system shall allow users to upload and read PDF documents within the app. The viewer shall support text highlighting, bookmarks, and annotations.
3.10 Template Community
FR-16: Template Sharing
The system shall provide a community marketplace where users can publish their tracker templates, browse templates shared by others, and clone a template to their own account.
Templates shall have ratings, tags, and download counts.
Users must have a verified account to publish templates.
3.11 AI Chatbot
FR-17: Conversational Assistant
The system shall include an AI chatbot accessible from any screen. The chatbot shall answer questions about the user's data, suggest tracker improvements, and execute commands (e.g., 'Add a hydration tracker').
3.12 Themes
FR-18: Theme System
The system shall support full UI theme switching including at minimum: Gaming Theme (dark, neon, game-inspired UI), Professional Theme (clean, minimal, business-appropriate), and Light/Dark defaults.
Theme shall optionally apply to timer appearance, notification style, and sound scheme.
3.13 Enterprise Features
FR-19: Group Account Management
Enterprise admins shall be able to create group accounts, invite members, and assign shared tracker templates to the entire group.
FR-20: Group Dashboard
Admins shall have access to an aggregated dashboard showing group-wide progress on shared goals without exposing individual member personal data.

Phase 3: Design
Software Design Description (SDD) — IEEE Std 1016
Document 03 of 07  |  LifeCore AI Life Tracker
1. Design Overview
1.1 Architectural Pattern
LifeCore uses a Client-Server architecture with a microservices-adjacent backend. The frontend is a React Native application connecting to a Node.js/Express REST API backed by PostgreSQL (via Supabase). AI features are handled through adapter services that interface with external LLM and vision APIs.
1.2 System Context Viewpoint
External actors and systems that interact with LifeCore:
End User (Mobile / Web) — the primary actor
Google Calendar API — bidirectional event sync
Anthropic Claude / OpenAI GPT-4 — LLM for generative features
OpenAI Whisper — audio transcription
Google Cloud Vision — food photo analysis
Nutrition API (USDA FoodData Central) — macro data
Apple Push Notification Service / Firebase Cloud Messaging — push notifications
2. Data Design
2.1 Core Database Schema
Key entities and their relationships. Full SQL migrations are maintained in /db/migrations/.

4.1 LLM Adapter Pattern
All LLM calls go through a single AIAdapter service. This allows switching between Anthropic Claude, OpenAI, and other providers without changing application code.
4.2 Agentic Workflow Pattern
Multi-step AI operations (tracker generation, workout planning) follow this pattern:

Step 2: AI analyzes intent and generates plan (streamed to UI)
Step 3: System presents preview with estimated completion
Step 4: User confirms, modifies, or cancels
Step 5: System executes confirmed plan, saves to database
Step 6: User receives confirmation with edit options

6. Security Design
All API endpoints require JWT authentication (except /api/templates/community which is optionally public).
JWTs expire in 1 hour; refresh tokens rotate on each use and expire in 30 days.
Audio files and PDF uploads are stored in Supabase Storage with signed URL access (1-hour expiry).
AI prompts are sanitized to prevent prompt injection attacks.
Food photos are processed server-side and not stored permanently — only the nutritional result is saved.
Enterprise data is isolated by organization ID in RLS policies.

3. AI Integration Build Guide
3.1 One-Prompt Tracker Generator
This is the flagship feature. Build it in Sprint 3–4 with the following implementation steps:
Create POST /api/tracker/generate endpoint
Accept: { prompt: string, userId: string }
System prompt instructs Claude to analyze life goals and output structured JSON tracker config
Use generateStructured(prompt, TrackerConfigSchema) from AIAdapter
Stream the generation progress to the client using Server-Sent Events
Frontend renders a live preview of the tracker as sections stream in
User confirms → save to trackers and tracker_sections tables
User edits → re-run generation on the modified prompt or manual section edit
3.2 Food Photo Analysis
Accept image upload from mobile camera or photo library
Send to Google Cloud Vision API for object detection
Pass detected food items to LLM: 'Estimate macros for: [items]'
Return structured nutrition data: { items: [{name, calories, protein, carbs, fat}], total: {...} }
Display confidence indicator — items with low confidence are flagged for user correction
3.3 Agentic Workout Generator
Step 1: Parse user prompt for goals, duration, equipment, frequency
Step 2: LLM generates weekly plan JSON (days × exercises × sets/reps)
Step 3: Show full plan preview before saving
Step 4: User can approve, edit individual exercises, or regenerate
Step 5: Save to workouts table; auto-schedule workouts as calendar events

1. Test Plan (Master)
1.1 Scope
In Scope: All P0 and P1 functional requirements from the SRS, all non-functional requirements with measurable metrics, security and authentication flows, Google Calendar sync, all AI feature happy paths and fallback paths.
Out of Scope: Enterprise features (tested separately pre-enterprise launch), third-party API internal behavior, platform-specific UI rendering differences beyond critical breakpoints.
1.2 Test Approach
Unit Testing: Jest for utility functions, Zustand stores, and API service layers.
Integration Testing: Supertest for API endpoints, Supabase test client for DB operations.
End-to-End Testing: Detox (mobile) and Playwright (web) for critical user flows.
Manual Exploratory Testing: One full exploratory session per sprint.
AI Feature Testing: Human evaluation rubric for AI output quality.
1.3 Pass/Fail Criteria
PASS: Actual result matches expected result exactly, or within specified tolerance.
FAIL: Any deviation from expected result. All FAILs are logged as incidents.
BLOCK: Test cannot execute due to environment or dependency issue.
Overall release gate: 100% of P0 tests pass, ≥95% of P1 tests pass.
1.4 Suspension/Resumption Criteria
Testing is SUSPENDED if: More than 3 P0 bugs are open simultaneously, the build fails to install on test device, or core auth fails.
Testing RESUMES when: All open P0 bugs are fixed and verified, a new clean build is available.
